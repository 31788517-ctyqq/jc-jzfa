
PK模块 产品优化方案

### 一、优化目标

- **从“数值对比”升级为“决策辅助”**：不仅展示差异，更告诉用户哪场比赛更值得投注、哪场比赛风险高。
- **从“单一表格”升级为“多维可视化”**：利用图表、色阶、标签降低认知负担。
- **从“单场PK”升级为“组合推荐”**：自动生成串关建议，提升用户决策效率。

### 二、核心产品逻辑升级

#### 2.1 智能标签系统（自动为每场比赛生成）

基于四大维度数值和模型状态，系统自动为每场比赛打上以下标签（可多选）：

| 标签类型 | 触发条件 | 示例 |
|---------|----------|------|
| 🔥 **绝对优势** | 综合实力 ≥ 25% | 主队绝对优势 |
| ⚠️ **模型打架** | 进球预测触发熔断 | 模型严重分歧，谨慎 |
| 💰 **过热风险** | 冷热指数 ≥ 1.40 | 市场过热，防冷 |
| 🧊 **冷门潜质** | 冷热指数 ≤ 0.85 | 逆向冷藏，关注 |
| 🎯 **实力均衡** | 综合实力在 -8% ~ +8% | 双方均衡，双选 |
| 🛡️ **防守大战** | 攻守实力 > 0.15 且 进球期望 < 2.0 | 小球格局 |
| ⚡ **对攻大战** | 攻守实力 > 0.15 且 进球期望 > 3.0 | 大球格局 |
| ✅ **高共识** | 胜平负离散度 < 7.0 | 机构分歧极小 |

#### 2.2 多场PK的组合推荐引擎

当用户勾选2-4场比赛时，系统自动计算三种推荐组合：

| 组合类型 | 算法逻辑 | 输出示例 |
|---------|----------|----------|
| **正路组合** | 选取“综合实力”最高的2-3场，且模型无熔断、热度不过热 | 利物浦(主胜) + 曼城(主胜) |
| **博冷组合** | 选取“冷热指数”低于0.85或“模型打架”的场次，搭配高赔选项 | 切尔西(客胜) + 横滨(平) |
| **双选容错组合** | 选取“实力均衡”的场次，推荐胜/平或平/负双选 | 鹿岛(胜/平) + 纽卡(负/平) |

#### 2.3 动态合计面板增强

原有底部合计面板（绝对值之和）保留，同时新增以下统计：

| 统计项 | 计算方式 | 业务价值 |
|-------|---------|----------|
| **平均综合实力** | 勾选场次的综合实力平均值 | 判断整体选场正路程度 |
| **优势一致性** | 主队优势场次 / 总场次 | 判断是否全部偏向主队 |
| **模型健康度** | 熔断场次数 / 总场次 | 风控提示 |
| **热度偏离度** | 冷热指数高于1.2或低于0.8的场次数 | 提示市场异常 |

### 三、UI/UX展示方案

#### 3.1 默认视图：横向雷达对比图（替代纯表格首屏）

- **X轴**：勾选的比赛（最多5场）
- **Y轴**：四项实力维度（归一化至-1~1）
- **图形**：分组柱状图 或 雷达图堆叠
- **色阶**：正值绿色（越长越优），负值红色（越长越劣）
- **交互**：鼠标悬停显示具体数值和标签

> 示例：  
> `利物浦`：净胜球 ████████ 0.35 | 综合实力 ██████ 0.22  
> `切尔西`：净胜球 ███ -0.12 | 综合实力 ██ -0.08  

#### 3.2 辅助视图：智能结论卡（位于图表右侧或下方）

固定区域展示以下卡片：

| 卡片标题 | 内容 |
|---------|------|
| 🎯 **最佳正路场次** | 综合实力最高的比赛 + 推荐方向 |
| ⚡ **最值博冷场次** | 冷热指数最低的比赛 + 推荐冷门方向 |
| 🔒 **最佳双选场次** | 实力最均衡的比赛 + 推荐双选 |
| 🧠 **组合推荐** | 正路串 / 博冷串 / 双选串，附带预计SP范围 |
| ⚠️ **风险提示** | 熔断/过热/高离散场次列表 |

#### 3.3 详细数据表格（折叠或分页）

保留原有表格，用于专业用户查看具体数值。支持点击表头排序，支持筛选（如只看主队优势场次）。

### 四、前端开发要点

| 功能点 | 实现说明 |
|-------|----------|
| 图表库 | 使用 ECharts 或 AntV 实现分组柱状图/雷达图 |
| 标签系统 | 后端返回每个场次的 `tags` 数组，前端直接渲染 |
| 组合推荐 | 后端根据简单规则计算，返回3组推荐串 |
| 合计面板增强 | 后端增加 `summary` 对象，包含平均综合实力、优势一致性等 |
| 响应式 | 移动端默认表格，平板以上默认图表 |

### 五、数据接口扩展建议

在原有 PK 接口响应中增加以下字段：

```json
{
  "matches": [...],
  "summary": {
    "avgCompositeStrength": 0.14,
    "homeAdvantageRatio": 0.75,
    "modelConflictCount": 1,
    "overheatCount": 0
  },
  "recommendations": {
    "positiveCombo": ["lineId_1", "lineId_3"],
    "coldCombo": ["lineId_2"],
    "doubleChanceCombo": ["lineId_4"]
  }
}
```

---

此方案在保持后端计算负担较小的前提下，大幅提升了前端信息的可读性和决策价值。
