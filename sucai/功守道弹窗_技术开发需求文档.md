## 1. 项目目标

将现有的**功守道量化弹窗**从占位假数据改造为**真实算法驱动**，完成从"数据获取 → 算法计算 → 弹窗展示"的完整闭环。

| 当前状态 | 目标状态 |
|---|---|
| 前端弹窗 UI 已就绪，但展示硬编码假数据 | 展示真实量化计算结果 |
| 服务端 `gongshoudao` 端点返回占位 JSON | 服务端从 API 获取 34 维统计字段并执行 7 阶段算法 |
| `server/data.json` 仅存储基础比赛信息 | 引入 `m.100qiu.com/api/dcListBasic` 外部数据源 |
| `sucai/gongshoudao/` 文件夹为空 | 存放核心算法模块 |

## 2. 数据来源与取值方案

### 2.1 数据接口

```
GET https://m.100qiu.com/api/dcListBasic?dateTime=26058
```

- `dateTime` 参数：`26058` = `2026-05-26` 的号码格式（`26`=年 `05`=月 `8`=日编号? 或直接日期编码）
- 每天登录时获取当天所有比赛的统计数据（单次请求即可拿到全量）
- 建议**按天缓存**，减少外部请求

## 3. 算法管道（7 阶段）

### 3.1 总体流程

```
                               ┌─────────────────────┐
   API 数据（34字段）           │  功守道 V24.0 引擎   │
   dcListBasic                 │  量化计算层          │
        │                      │                     │
        ▼                      │ 字段映射 → 计算 → 输出│
  ┌───────────┐                │                     │
  │ 字段解析  │──────────────▶│  P0: 实力分析        │
  │ 正则提取  │                │  P1: 大小球分析      │
  └───────────┘                │  P2: 净胜球分析      │
                               │  P3: 比分矩阵预测    │
                               └─────────┬───────────┘
                                         │
                                         ▼
                                  ┌─────────────┐
                                  │ 弹窗渲染     │
                                  │ preview/preview│
                                  │ miniprogram   │
                                  └─────────────┘
```

### 3.2 各阶段输入/输出

| 阶段 | 名称 | 核心输入 | 产出 | 优先级 |
|---|---|---|---|---|
| **第一阶段** | 字段映射与解析 | API 原始 JSON | 34个标准变量（已对齐） | P0 |
| **第二阶段** | 阵之第一维·量化盘面 | 赢球差分布 + 进球分布 + 攻防效率 | ①进攻优势度 ②防守优势度 ③综合攻守优势 ④胜平负交叉分布 | P0 |
| **第三阶段** | 实力阶梯映射 | 综合攻守优势 `P_total` | 7档实力标签 + 边界锁 `Δ_bound` | P0 |
| **第四阶段** | 阵之第二维·大小球博弈 | 主客权重 + 场地烈度 + 大球率 | ①总进球期望 ②总球数弹性区间 ③主客预期进球 `xG` | P1 |
| **第五阶段** | 阵之第三维·让球分析 | `xG_home/xG_away` + 10场净胜球 + `Total_战` | ①预期净胜球差 ②Anchor锚点 ③7场通过判定 ④三者一致裁决 | P1 |
| **第六阶段** | 比分矩阵合围 | `xG` + 三重物理锁 + 历史数据 | TOP8 比分 + 概率 | P2 |
| **第七阶段** | 报告排版 | 前六阶段全量输出 | 结构化前景数据 | P2 |

### 3.3 ⚠️ 公式缺口说明

`ZUQIU.md`（原始协议）中部分公式为**自然语言描述，尚未转化为精确数学公式**，需要在开发时按阶段补充：

| 阶段 | 可由描述直接编码 | 需要补充/推导的公式 |
|---|---|---|
| 第一阶段 | ✅ 全部（读字段 + 正则） | 无 |
| 第二阶段 | 部分（赢球格局对冲、进球厚度分布） | ①进攻权重切分公式 ②格局状态机精确判定阈值 |
| 第三阶段 | ✅ 全部（7档区间判定） | 无 |
| 第四阶段 | 部分（权重、场地烈度） | ①总进球期望精确公式 ②攻防次数还原公式 ③呼吸权重公式 |
| 第五阶段 | 部分（7场阈值统计逻辑） | ①`Total_战` 精确公式 ②Anchor 阈值 |
| 第六阶段 | 部分（三重锁逻辑） | ①泊松矩阵公式 ②十字对冲因子 ③大盘杠杆因子 |
| 第七阶段 | ✅ 全部（排版逻辑） | 无 |

**策略：优先级 P0 采用保守估算公式先行落地，P1/P2 逐步补齐。**

## 5. 技术实现路径

### 5.1 架构方案对比

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| **A: 服务端实时计算** | 前端调用 `/api` → 服务端实时取 API + 计算 | 数据最新，逻辑集中在后端 | 外部 API 延迟风险，高并发压力 |
| **B: 服务端预计算+缓存** | 爬虫/定时任务提前从 API 拉取并计算好 | 响应快，可离线使用 | 数据非实时 |
| **C: 前端直接计算** | 前端直接调 `dcListBasic` 并在客户端运行算法 | 无后端压力 | 暴露 API key，计算量大，不适合移动端 |

**推荐方案：A+B 混合**

```
定时任务（每30分钟）
  │
  ▼
dcListBasic API ──▶ 缓存 stats_bank.json ──▶ 计算引擎 ──▶ 缓存 gs_cache.json
                                                              │
                                                              ▼
用户点击弹窗 ──▶ /api gongshoudao ──▶ 读 gs_cache ──▶ 返回计算结果
```

层级关系：
1. **数据层**：`dcListBasic` API → `stats_bank.json`（原始统计，按日期分）
2. **计算层**：`gongshoudao/core/` → 读 `stats_bank` → 按 `matchId` 计算 → `gs_cache.json`
3. **接口层**：`server/index.js` 的 `gongshoudao` case → 读缓存返回
4. **展示层**：`gongshoudao.js` + `index.html`（现有，微调字段名即可）

## 7. 风险与依赖

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| `dcListBasic` API 不可用 | 全链路中断 | 缓存最新一份数据，降级显示"暂无最新数据" |
| 算法公式缺失（部分未定义精确公式） | 计算结果不准 | P0 阶段采用保守估算公式，后续迭代优化 |
| 比赛数量多时计算耗时 | 弹窗响应慢 | 预计算 + 缓存，不在请求链路中实时计算 |
| 小程序端样式适配 | 显示错乱 | 复用 preview 端样式，rpx 适配 |

---

## 8. 验收标准

1. **数据完整**：每天的比赛能从 `dcListBasic` 正确拉取全量 34 个统计字段
2. **算法可跑**：选择 1 场已知比赛（如玛丽港 vs 拉赫蒂），手动验算每个阶段的中间值是否能由代码输出
3. **弹窗可用**：点击比赛卡片"功守道量化"按钮后 1 秒内弹出，展示真实计算数据（非占位假数据）
4. **边界安全**：极端数据（全是 0、全胜、全败）不会导致页面崩溃或 NaN
5. **小程序同步**：miniprogram 端也能触发功守道弹窗
